# Handoff — April 30, 2026: Dev Server `Permission denied` Fix

## Summary

- `node_modules/.bin/vite` had mode `644` (no exec bit) → `sh: 1: vite: Permission denied` when running `npm run dev`.
- **Cause:** a previous `npm install` ran from Windows / PowerShell against the `\\wsl.localhost\...` path, which strips Unix executable bits from `node_modules/.bin/*`.
- **Fix:** nuked `node_modules` + `package-lock.json` and reinstalled inside WSL — `vite` is now a proper symlink (`node_modules/.bin/vite -> ../vite/bin/vite.js`).
- Added a troubleshooting note to [README.md](README.md) under the Quick Start "Important" callout so future-you doesn't have to re-diagnose this.

## Repro of the fix

From a WSL terminal:

```bash
cd /home/kerry/programming/portfolio
rm -rf node_modules package-lock.json
source ~/.nvm/nvm.sh && nvm use 22
npm install
npm run dev -- --host --port 4000
# → http://localhost:4000
```

## How to avoid this in the future

- **Always** run `npm install` from inside WSL (a WSL terminal, or wrapped via `wsl -e bash -ic '...'` from PowerShell as shown in README "Option B").
- **Never** run `npm install` in raw PowerShell while `cd`'d into `\\wsl.localhost\Ubuntu\home\...`. The Windows npm will install into the WSL filesystem but strip the Unix `+x` bit on every binary symlink target, breaking every CLI in `node_modules/.bin/`.
- Symptom to recognize next time: `Permission denied` on any `node_modules/.bin/*` script. Confirm with `ls -la node_modules/.bin/vite` — if it shows `-rw-r--r--` instead of a symlink, that's the bug.
